  C HARNESS FOR EXACT KAT MATCHING
════════════════════════════════════════════════════════════════════════
  Written: ml_kem_kat.c

  Compile & run (WSL / Linux):
    sudo apt install libssl-dev
    gcc ml_kem_kat.c -loqs -lssl -lcrypto -o ml_kem_kat
    ./ml_kem_kat

  Why C for exact KAT matching?
  • Python libs use OS entropy → random keys each run
  • NIST KAT requires a seeded AES-CTR DRBG (C only)
  • C harness feeds the 48-byte seed → deterministic keys
  • liboqs test suite uses this approach internally

════════════════════════════════════════════════════════════════════════
  VIVA REFERENCE — WHAT EACH CHECK MEANS
════════════════════════════════════════════════════════════════════════

  pk_size / sk_size / ct_size / ss_size
    → Verifies the implementation produces outputs of exactly the sizes
      mandated by NIST FIPS 203 Table 2.  A wrong size = broken impl.

  round_trip
    → The most critical functional test: the shared secret produced by
      encapsulation (using public key) must exactly equal the shared
      secret produced by decapsulation (using secret key + ciphertext).
      This confirms correctness of the KEM.

  pk/ct/ss_not_zero
    → Guards against degenerate all-zero outputs (RNG failure).

  kat_ss_note  (INFO, not FAIL)
    → Python libraries use the OS random source so keys differ every
      run.  Exact KAT SS matching requires feeding the 48-byte NIST
      seed into an AES-CTR DRBG — this is only possible in C.
      The C harness (ml_kem_kat.c) performs that exact check.

  NIST KAT seed (48 bytes)
    → The NIST KAT generation script initialises a deterministic DRBG
      with this seed, calls keypair + encaps, and records pk, sk, ct,
      ss.  Any conforming implementation fed the same seed must produce
      the identical outputs.
